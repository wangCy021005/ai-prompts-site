# 代码审查提示词

> **用途**：提 PR 前让 AI 帮你审一遍，找出潜在 bug、安全风险、性能问题，给出优化建议。
> **适用**：写完功能自查、接手他人代码review、重构后回归检查。

## 提示词

```
你是一位资深 [语言/技术栈] 工程师，请审查以下代码变更。

【上下文】
- 项目：[一句话描述项目，如"一个订单后台服务，Go + MySQL"]
- 这次的改动目的：[一句话，如"新增批量退款接口"]
- 已知约束：[如"QPS 约 500，退款金额必须事务保证"]

【代码变更】
[粘贴 diff 或完整代码片段]

请从以下维度审查，按严重程度排序输出：

1. **Bug 风险**（最高优先级）
   - 空指针 / 边界条件 / 并发问题
   - 事务、锁、资源泄漏
   - 错误处理是否完整（吞异常、漏 return）

2. **安全风险**
   - SQL 注入、XSS、权限校验缺失
   - 敏感信息日志泄漏

3. **性能问题**
   - N+1 查询、循环里调外部接口
   - 不必要的全量加载、缺索引的查询

4. **可维护性**
   - 命名、函数长度、重复代码
   - 是否符合该语言惯用法

输出格式：
- 🔴 严重问题（必须修）：文件:行号 — 问题 — 建议改法
- 🟡 建议优化（可选）：同上
- 🟢 写得不错的地方（一句话带过即可）

不要泛泛而谈，每条都指向具体行号和代码。没有问题的维度直接跳过，不要凑数。
```

## 使用说明

1. `[语言/技术栈]` 填 Go / Java / TypeScript / Python 等，让 AI 用对应惯用法审
2. **改动目的必须写清楚**——AI 不知道你想干嘛，就无法判断"是不是写复杂了"
3. 贴 diff 比贴完整文件效果好，AI 能聚焦在变更上；但如果是新文件就贴完整
4. 已知约束很重要：不写约束 AI 会用通用标准审，可能误报一堆"理论上可能"的问题

## 分场景变体

### 场景一：快速自查（赶时间）

```
帮我快速 review 这段 [语言] 代码，只看 Bug 和安全风险，其他不用管。
[粘贴代码]
```

### 场景二：接手老代码 review

```
我刚接手这段代码，请帮我审查并解释它在做什么、有哪些坑。
重点：1) 这段代码的业务意图  2) 历史遗留风险  3) 哪些地方以后改要小心
[粘贴代码]
```

### 场景三：并发/事务专项

```
这段代码涉及并发和数据库事务，请重点检查：
1. 竞态条件（goroutine/线程安全）
2. 事务边界和隔离级别
3. 死锁可能性
4. 幂等性
[粘贴代码]
```

## 调试技巧

| 问题 | 解决 |
|------|------|
| AI 报了一堆"理论上可能"的问题 | 在已知约束里写清楚 QPS、数据规模、并发量 |
| AI 漏掉了真实 bug | 把单元测试 / 调用方代码也贴上，给它上下文 |
| AI 建议的改法不符合项目风格 | 贴一段项目里现有的"标准写法"作为参考 |
| 输出太长抓不住重点 | 加一句"只输出 🔴 严重问题，最多 5 条" |
| AI 一直在夸代码写得好 | 加一句"假设这段代码一定有 bug，努力找出来" |

## 效果示例

输入一段有问题的 Go 代码（示意）：

```go
func Transfer(from, to int, amount float64) error {
    balance := getBalance(from)
    if balance < amount {
        return errors.New("余额不足")
    }
    updateBalance(from, -amount)
    updateBalance(to, amount)
    return nil
}
```

AI 输出（节选）：

```
🔴 严重问题：
- 余额检查与扣款之间没有事务，存在 TOCTOU 竞态：并发下两人同时转账
  可绕过余额检查。建议：把 getBalance + updateBalance 包进同一个事务，
  并用 SELECT ... FOR UPDATE 锁行。
- updateBalance(to, amount) 如果失败，from 账户已扣款，钱会丢。
  建议：事务内执行，失败回滚；或使用补偿机制。

🟡 建议优化：
- amount 用 float64 存金额有精度风险，建议用 int 分或 decimal。
- 错误信息太笼统，建议返回带错误码的 sentinel error，方便上层判断。
```

这种带行号、带根因、带改法的输出，比单纯说"有并发问题"有用得多。

## 注意事项

⚠️ **AI 审完不等于可以不让人 review**：AI 擅长找模式化问题（空指针、SQL 注入），但业务逻辑正确性还得靠人。

⚠️ **不要贴敏感信息**：密钥、内网域名、真实用户数据脱敏后再贴。

⚠️ **大文件分段审**：超过 800 行的文件按模块拆开分别审，整文件糊上去 AI 会漏问题。
