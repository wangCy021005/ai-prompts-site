# Bug 定位提示词

> **用途**：线上报警 / 测试报错时，把现象和日志丢给 AI，让它帮你定位根因、给出排查方向。
> **适用**：线上报警排查、复现不稳定的 bug、看不懂的报错栈、偶现问题。

## 提示词

```
你是一位资深 [语言/技术栈] 工程师，帮我定位一个 bug。

【现象】
- 表现：[一句话描述，如"线上偶尔返回 500，接口 /api/order/create"]
- 频率：[如"每天 3-5 次，集中在晚高峰"]
- 影响范围：[如"约 0.1% 请求受影响，用户重试通常成功"]
- 开始时间：[如"7 月 1 号上线后开始出现"]

【报错信息】
[粘贴完整报错栈 / 日志，不要截断]

【相关代码】
[粘贴最可能相关的代码片段，比如报错栈指向的函数]

【已排查过的】
[列出你已经确认没问题的，如"数据库连接池正常、依赖服务正常、机器资源充足"]

请输出：
1. **最可能的根因 Top 3**（按可能性排序），每个根因说明：
   - 为什么会导致这个现象
   - 如何进一步验证（看什么日志、加什么监控、写什么复现脚本）
2. **快速验证步骤**：从最快验证的根因开始排
3. **临时止血方案**（如果根因确认耗时较长）
4. **长期修复建议**

不要给一个答案就完事，给多个假设 + 验证路径，让我自己挑能快速验证的。
```

## 使用说明

1. **现象描述要精确**：不要写"系统挂了"，要写"哪个接口、什么场景、错误码、频率"
2. **报错栈一定要完整**：不要只贴最后一行，AI 要看完整调用链才能定位
3. **"已排查过的"很关键**：能省掉 AI 给一堆你已经试过的建议
4. **频率和开始时间**能极大缩小范围：上线后开始出现 → 大概率是这次改动引入

## 分场景变体

### 场景一：偶现 / 难复现的 bug

```
这个 bug 很难复现，请帮我分析可能的触发条件。
现象：[描述]
报错：[粘贴]
代码：[粘贴]

重点分析：
1. 哪些并发场景 / 时序可能触发
2. 哪些边界数据可能触发（空值、超长、特殊字符）
3. 给我一个能强制复现的测试用例思路
```

### 场景二：性能问题（慢 / 超时）

```
接口偶发超时，帮我定位。
- 接口：[名称]
- 正常 RT：[如 50ms]，异常 RT：[如 5s]
- 发生时机：[如"每天 10 点准时变慢"]
- 该接口涉及的调用链：[粘贴或描述]

请从以下角度分析：
1. 哪一层最可能慢（DB / 缓存 / 下游 RPC / GC / 锁）
2. 怎么打点定位到具体那一层
3. 给我一个排查 checklist
```

### 场景三：看不懂的报错栈

```
这段报错栈我看不懂，请逐行解释并定位根因：
[粘贴完整报错栈]
项目技术栈：[如"Spring Boot 2.7 + MyBatis + MySQL"]
```

## 调试技巧

| 问题 | 解决 |
|------|------|
| AI 一直往一个方向猜 | 加"给 Top 3 假设，不要只给一个" |
| AI 建议的验证步骤没法落地 | 加"验证步骤必须具体到命令/SQL/代码片段" |
| AI 完全跑偏 | 你贴的代码可能不是真凶，把报错栈里更上层的代码也贴上 |
| 现象描述太模糊 AI 没法分析 | 先补"频率/时间/触发条件"三要素再问 |
| 偶现 bug AI 说"无法定位" | 让它"列出所有可能的竞态条件"，它会列得很全 |

## 进阶：让 AI 帮你写复现脚本

定位完根因，紧接着用这个提示词逼出复现用例：

```
基于上面的根因假设 [选一个]，帮我写一个能稳定复现的测试用例。
要求：
1. 用 [语言] 的测试框架
2. 模拟并发 / 边界数据，强制触发该 bug
3. 加注释说明每一步在模拟什么场景
4. 最后给出断言：什么样说明 bug 被触发了
```

复现脚本一旦写出来，bug 基本就修了一半。

## 效果示例

输入现象：

```
现象：订单创建接口偶尔返回 500
报错：sql: database is closed
频率：每天晚高峰 10-20 次
开始时间：上线增加批量查询后出现
```

AI 输出（节选）：

```
Top 3 根因：
1. 【最可能】数据库连接池配置过小 + 批量查询占用连接过久
   - 现象解释：晚高峰 QPS 高，批量查询持有连接时间长，新请求拿不到连接，
     超时后连接被回收，但请求仍在用 → "database is closed"
   - 验证：监控连接池 active/wait 指标，看晚高峰是否打满
   - 复现：压测批量查询接口，观察连接池水位

2. 【次可能】批量查询存在连接泄漏（rows 未 Close）
   - 验证：grep 代码里 *sql.Rows 是否都有 defer Close
   - 复现：循环调用批量接口 N 次，看连接数是否单调上升

3. 【可能性低】数据库服务端主动断连，客户端未感知
   - 验证：查 MySQL wait_timeout / 服务端错误日志

临时止血：把连接池 MaxOpenConns 调大 + 给批量查询接口加超时熔断。
长期修复：修复可能的连接泄漏 + 批量查询改成分页游标。
```

这种"多个假设 + 验证路径 + 止血方案"的输出，比直接给一个答案实用 10 倍。

## 注意事项

⚠️ **AI 给的是假设不是结论**：线上 bug 最终一定要靠日志、监控、断点确认，不要凭 AI 一句话就改代码。

⚠️ **不要贴真实用户数据**：手机号、订单号、身份证全部脱敏。

⚠️ **复杂调用链要画清楚**：微服务之间的调用关系，AI 不画图就只能瞎猜，最好用文字描述清楚"A 调 B，B 调 C，C 报错"。
