# 代码重构提示词

> **用途**：接手屎山代码 / 函数太长难维护 / 性能瓶颈，让 AI 给重构方案 + 改造步骤。
> **适用**：还技术债、性能优化、长函数拆分、消除重复代码、老代码现代化。

## 提示词

```
你是一位资深 [语言/技术栈] 工程师，请帮我重构以下代码。

【重构目标】（选 1-2 个，别贪多）
- [ ] 提升可读性（命名、结构、职责分离）
- [ ] 提升性能（减少冗余调用、优化数据结构）
- [ ] 提升可测试性（解耦依赖、便于 mock）
- [ ] 消除重复（DRY）

【约束】
- 不能改变外部行为（输入输出契约不变）
- 不能引入新依赖：[如"只用标准库"]
- 必须保留向后兼容：[如"旧 API 签名不能改"]
- 性能基线：[如"当前 P99 200ms，重构后不能劣化"]

【代码】
[粘贴要重构的代码]

请输出：
1. **现状诊断**：这段代码的核心问题（最多 3 条，按严重程度排序）
2. **重构方案**：
   - 整体思路（一段话讲清楚怎么改）
   - 关键改动点（每点对应一个具体问题）
   - 重构后的完整代码
3. **改动风险**：哪些地方最容易改出 bug，需要重点回归
4. **分步实施计划**：如果改动较大，拆成几个可独立提交的 step

⚠️ 不要为了重构而重构：能局部改好的别推倒重写。每条改动都要说清"解决了什么问题"。
```

## 使用说明

1. **重构目标只选 1-2 个**：全选会让 AI 改得很激进，反而引入风险
2. **约束必须写死**：尤其是"不能改外部行为"和"不能引新依赖"，否则 AI 会给你引入一堆新框架
3. **性能基线很重要**：没基线 AI 不知道性能优化算不算成功
4. **分步实施计划是精华**：大重构一次性提交是灾难，拆成小步能随时回滚

## 分场景变体

### 场景一：长函数拆分

```
这个函数太长（[X] 行），帮我拆成职责单一的小函数。
要求：
1. 每个函数不超过 30 行
2. 函数名要能自解释，看名字就知道干嘛
3. 先给出函数拆分后的结构（伪代码），我确认后再写实现
4. 标注哪些部分是可独立提取的纯函数（方便单测）
[粘贴代码]
```

### 场景二：性能优化专项

```
这段代码是性能瓶颈，帮我优化。
- 当前耗时：[如"10000 条数据处理需要 8s"]
- 目标：[如"降到 1s 以内"]
- 瓶颈点（已 profile）：[如"80% 时间花在循环里的 DB 查询"]

请：
1. 分析性能瓶颈根因
2. 给出 2-3 个优化方向，按收益/成本比排序
3. 推荐方案给出完整实现
4. 说明优化后预期性能 + 可能的副作用
[粘贴代码]
```

### 场景三：消除重复代码

```
这几段代码逻辑高度重复，帮我抽取通用逻辑。
要求：
1. 抽取出通用的函数/类，保留各处差异点作为参数
2. 不要过度抽象——只抽象真正重复的部分，差异点保持显式
3. 给出抽取后的调用方代码，确认调用更简洁了
[粘贴多段重复代码]
```

### 场景四：老代码现代化

```
这段 [老语言版本/老框架] 代码想升级到 [新版本]，帮我重构。
- 当前版本：[如"Python 2.7"]
- 目标版本：[如"Python 3.11"]
- 升级范围：[如"只升级语法，不改架构"]
请重点利用新版本的特性（type hint、dataclass、match-case 等），
但保持外部接口不变。
[粘贴代码]
```

## 调试技巧

| 问题 | 解决 |
|------|------|
| AI 改得太激进，引入新风险 | 加"采用最小改动原则，能局部改的别整体重写" |
| AI 引入了一堆新依赖 | 加"只能用标准库 / 已有依赖，新增依赖需单独说明理由" |
| 重构后行为变了 | 加"先列出所有外部行为契约，重构后逐一确认未变" |
| 改动太大不敢合 | 加"拆成可独立提交的 step，每步都能编译通过且有测试覆盖" |
| AI 改完反而更难懂 | 加"重构后代码要更易读，如果变得更绕就放弃这次重构" |

## 进阶：重构前后对比验证

让 AI 自己证明重构没改坏：

```
重构完成后，请额外输出：
1. **行为等价性证明**：列出重构前后输入输出的对照表（至少 3 组测试用例），
   证明核心逻辑等价
2. **回归测试用例**：给我 5 个能覆盖关键路径的测试用例，
   重构前后跑这些用例结果应该一致
3. **需要人工确认的边界**：哪些地方行为可能有细微差异，需要我重点验证
```

老代码没测试覆盖时，这一步能救命。

## 效果示例

输入一个又长又杂的函数（示意）：

```python
def process_orders(orders):
    result = []
    for o in orders:
        if o['status'] == 'paid':
            if o['amount'] > 1000:
                discount = o['amount'] * 0.1
            else:
                discount = 0
            final = o['amount'] - discount
            if o['vip']:
                final = final * 0.95
            o['final_amount'] = final
            db.save(o)
            send_email(o['email'], final)
            result.append(o)
    return result
```

AI 输出诊断 + 重构（节选）：

```
【现状诊断】
1. 函数混了 4 个职责：过滤、计价、持久化、通知 → 难测试难复用
2. 计价逻辑硬编码，新增规则要改这里 → 违反开闭原则
3. 副作用（db.save / send_email）和纯逻辑耦合 → 无法单测计价

【重构方案】
拆成：filter_paid / calculate_price / persist / notify 四个函数，
计价逻辑独立成纯函数，副作用在外层编排。

def process_orders(orders):
    paid = filter_paid(orders)
    for o in paid:
        o['final_amount'] = calculate_price(o)
    persist(paid)
    notify_all(paid)
    return paid

def calculate_price(o):  # 纯函数，可单测
    discount = o['amount'] * 0.1 if o['amount'] > 1000 else 0
    final = o['amount'] - discount
    return final * 0.95 if o['vip'] else final

【改动风险】
- calculate_price 抽出后，原代码里 o['final_amount'] 的写入时机变了，
  需确认下游没有依赖"save 之前 final_amount 已写入"的隐式顺序
【分步实施】
Step 1: 抽出 calculate_price（纯函数，低风险）
Step 2: 抽出 filter_paid（低风险）
Step 3: 拆分副作用 persist / notify（中风险，需回归邮件发送）
```

## 注意事项

⚠️ **没有测试覆盖的老代码慎重重构**：先补测试再重构，否则改出 bug 都不知道。

⚠️ **大重构必须分步提交**：每个 step 独立可回滚，别攒一个大 PR。

⚠️ **AI 给的方案要复核"行为等价性"**：AI 有时会"顺手优化"掉它认为没用的逻辑，那可能是别人故意写的 hack。
