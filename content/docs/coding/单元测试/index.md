# 单元测试生成提示词

> **用途**：写完功能让 AI 帮你生成单元测试，覆盖正常路径、边界条件、异常分支，提测前补测试保平安。
> **适用**：新功能补单测、老代码补回归测试、提高覆盖率、TDD 写测试先行。

## 提示词

```
你是一位注重测试质量的 [语言/技术栈] 工程师，请为以下代码生成单元测试。

【被测代码】
[粘贴要测试的函数/类]

【测试框架】[如"Go: testify / Python: pytest / Java: JUnit5"]
【Mock 库】[如"gomock / unittest.mock / Mockito"]
【约束】
- 测试不能依赖外部服务（DB / Redis / 下游 RPC 全部 mock）
- 每个测试用例独立，不能有执行顺序依赖
- 覆盖率目标：[如"语句覆盖 90%+，重点分支必覆盖"]

请按以下分类输出测试用例：

1. **正常路径**（Happy Path）
   - 最常见的输入，验证主流程正确

2. **边界条件**
   - 空值 / 零值 / 最大值 / 最小值
   - 空集合 / 单元素 / 大集合
   - 临界值（如 amount = 0, amount = 余额）

3. **异常分支**
   - 依赖服务失败（DB 超时、RPC 错误、返回异常）
   - 非法输入（类型错误、越界、格式错）
   - 并发 / 超时场景（如适用）

4. **业务规则**
   - 涉及业务逻辑的特殊判断（如"VIP 用户打折"、"金额超限拦截"）

每个用例要求：
- 用 Given-When-Then 结构（或 Arrange-Act-Assert）
- 测试函数名能自解释：test_[场景]_[预期结果]
- 断言要具体，不要只断言"不报错"
- mock 设置要最小化，只 mock 这次用到的依赖

输出完整的可运行测试代码，并在每个用例上方用一行注释说明它在测什么。
```

## 使用说明

1. **测试框架和 Mock 库必须明确**：否则 AI 用默认的，可能跟项目风格不一致
2. **被测代码要完整**：贴函数 + 它依赖的接口定义，AI 才能正确 mock
3. **覆盖率目标写实**：写"100%"AI 会硬凑无用用例，写"重点分支必覆盖"更实用
4. **业务规则那栏很关键**：纯技术测试 AI 写得很好，业务规则得你告诉它

## 分场景变体

### 场景一：TDD 测试先行

```
我要实现一个 [功能描述] 的函数，请先帮我写测试用例，再写实现。
函数签名：[如"def refund(order_id: str, amount: int) -> RefundResult"]
业务规则：
1. [如"订单必须是已支付状态"]
2. [如"退款金额不能超过订单金额"]
3. [如"同一订单 5 分钟内只能退一次"]

请：
1. 先列出所有测试场景（表格形式：场景 / 输入 / 预期）
2. 我确认后生成完整测试代码
3. 最后给出最小实现让测试通过
```

### 场景二：补老代码测试

```
这段老代码没有测试，帮我补单测。
难点：依赖很多外部服务，需要全部 mock。
请：
1. 先列出需要 mock 的依赖清单
2. 评估哪些代码不可测（比如静态方法直接 new 了依赖），标出来
3. 对可测部分生成测试用例
4. 不可测部分给出"为可测试性小幅重构"的建议
[粘贴代码]
```

### 场景三：提高覆盖率

```
当前覆盖率报告显示这几行没覆盖，帮我补测试：
[粘贴未覆盖的代码行 + 行号]

请：
1. 分析每行未覆盖的原因（缺什么输入才会走到）
2. 生成对应的测试用例
3. 标注哪些是"死代码"（永远走不到，可以删）
```

### 场景四：回归测试（修完 bug）

```
刚修了一个 bug：[描述 bug 现象和根因]
请帮我写回归测试，确保这个 bug 不会再出现。
要求：
1. 先写一个能复现原 bug 的测试（用修复前的逻辑应该失败）
2. 修复后该测试应该通过
3. 再补 2-3 个相关边界用例，防止类似 bug
[粘贴修复后的代码]
```

## 调试技巧

| 问题 | 解决 |
|------|------|
| AI 生成的测试跑不起来 | 贴一段项目里现有的测试代码作为"风格示例" |
| mock 写得太复杂 | 加"只 mock 这次用到的依赖，其他用真实实现" |
| 测试用例太泛，覆盖不到 bug | 加"针对每行代码想一个能走到它的输入" |
| AI 生成一堆重复用例 | 加"每个用例必须测一个独立的场景，去重" |
| 测试依赖执行顺序 | 加"每个测试独立 setup/teardown，禁止共享状态" |

## 进阶：让 AI 自评测试质量

生成完后追加这个提示词，让 AI 自己审一遍：

```
请对你刚生成的测试做自评：
1. **缺陷检测**：哪些用例断言太弱（只断言"不报错"算弱断言）？
2. **冗余检测**：哪些用例重复了，可以合并？
3. **遗漏检测**：被测代码里有没有完全没覆盖到的分支？
4. **脆弱性**：哪些用例依赖了实现细节，重构后会误报？
按问题严重程度输出修改建议。
```

AI 自评经常能发现自己漏掉的边界，比一次生成质量高很多。

## 效果示例

输入一个简单的计价函数（示意）：

```python
def calc_price(amount: int, is_vip: bool, coupon: float = 0) -> float:
    if amount < 0:
        raise ValueError("金额不能为负")
    if amount == 0:
        return 0.0
    price = float(amount)
    if amount > 1000:
        price *= 0.9
    if is_vip:
        price *= 0.95
    price -= coupon
    if price < 0:
        price = 0.0
    return price
```

AI 输出测试用例（节选）：

```python
import pytest

# 正常路径：普通用户小额订单，不打折不抵扣
def test_normal_small_order():
    assert calc_price(100, is_vip=False) == 100.0

# 边界：刚好超过 1000 享受折扣
def test_discount_boundary_just_above_1000():
    assert calc_price(1001, is_vip=False) == pytest.approx(900.9)

# 边界：刚好 1000 不打折
def test_discount_boundary_exactly_1000():
    assert calc_price(1000, is_vip=False) == 1000.0

# 业务规则：VIP + 大额，折扣叠加
def test_vip_large_order_stacked_discount():
    # 2000 * 0.9 * 0.95 = 1710
    assert calc_price(2000, is_vip=True) == pytest.approx(1710.0)

# 异常：负数金额
def test_negative_amount_raises():
    with pytest.raises(ValueError, match="金额不能为负"):
        calc_price(-1, is_vip=False)

# 边界：优惠券抵完为负，应归零
def test_coupon_exceeds_price_clamped_to_zero():
    assert calc_price(100, is_vip=False, coupon=200) == 0.0

# 边界：金额为 0 直接返回
def test_zero_amount_returns_zero():
    assert calc_price(0, is_vip=True) == 0.0
```

注意每个用例都覆盖了一个独立的判定分支，断言具体到数值，不是"assert result is not None"这种废话。

## 注意事项

⚠️ **AI 生成的测试不一定能直接跑通**：mock 的接口签名、import 路径经常要手动调，跑一次修一次。

⚠️ **别只追求覆盖率**：100% 覆盖率不代表测了所有场景，分支组合才是重点。

⚠️ **测业务规则 > 测实现细节**：测"VIP 打 95 折"是对的，测"内部用了 if-else"是脆弱测试，重构就挂。

⚠️ **mock 不要过度**：把所有东西都 mock 了，测试就变成"测 mock 而不是测代码"，没意义。
